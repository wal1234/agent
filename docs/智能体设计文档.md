### 3.2 缺陷查重 & 相似聚类模块 ⭐ P0

#### 3.2.1 真实数据驱动的聚类维度

基于已观察到的共性模式，聚类维度按价值排序：

| 维度 | 算法 | 真实样本验证 |
|------|------|-------------|
| **业务动作聚类** | 关键词匹配（如"复制新增"、"导出"、"反显"）| 能识别出 6 条复制类、7 条导出类 |
| **技术模式聚类** | 关键词 + 根因文本 TF-IDF | 能识别 4 条事务/锁问题、5 条越权 |
| 模块/服务聚类 | module 字段 + defect_name 关键词 | 大屏抽奖 6 条、积分账户 8 条 |
| 错误类型聚类 | problem_type 字段 | 代码逻辑错误 / 特殊场景未处理 占 50% |
| 时间窗聚类 | occurrence_time 周/月维度 | 1 月集中爆发抽奖类问题 |

#### 3.2.2 查重输出（真实数据示例）

```markdown
## 缺陷聚类报告（基于 2025 台账有效 80 条）

### 🔴 高价值集群 #1：复制新增功能系统性缺陷
- **共性根因**：复制时未清空字段 / 未重新校验 / 未重新赋值
- **典型缺陷**：DEF-0022、0033、0073、0074、0078、0079
- **责任端**：前端为主（6/6）
- **建议**：将"复制新增"列为前端代码 Review 必查项；增加复制行为自动化用例

### 🔴 高价值集群 #2：事务/分布式锁/异步并发
- **共性根因**：事务未提交 / @DistributedLock 时长不够 / select+update 非原子
- **典型缺陷**：DEF-0028、0053、0071、0075
- **责任端**：后端
- **建议**：制定《事务边界与锁使用规范》；增加并发场景压测

### 🔴 高价值集群 #3：安全漏洞——越权类
- **共性根因**：接口未校验 user 与请求资源归属
- **典型缺陷**：DEF-0008、0009、0010、0063（XSS DEF-0055 单独）
- **责任端**：后端
- **建议**：所有 query/edit 接口加入 @PermissionCheck 强制注解
```

#### 3.2.3 查重算法选型

**V1 采用混合方案：**
- **关键词命中**（O(1) 快速预筛）→ 用于"复制新增"、"越权"、"超时"等业务词典
- **Jaccard 相似度**（轻量）→ 标题 + 根因 + 模块的 token 集合
- **LLM 终判**（仅 Top K） → 给最终新/相似/复发判定

> 不直接上向量检索：80 条样本数据量不够，且关键词命中已能覆盖 80% 场景。后续数据量到 500+ 再升级。

### 5.1 责任界定（原延后原因，现已实现）

**当时不优先的现实理由**：
- `responsibility_ratio` 字段污染严重，27 条无效值
- 8+ 名责任人已离场，无法溯源

- 业务实际中已通过缺陷分析会评定，智能体重复劳动

**最终实现方案**（突破延后限制）：
- 责任比规范化器自动识别 Excel 时间格式污染（如 `01:00:00` → `1:0`）
- 离职人员风险卡片：自动识别 `dev_owner` 含"已离场"的缺陷并提示知识断层
- 漏测原因启发式规则（6 类规则，零 LLM 依赖）
- 批量责任画像：团队 / 人员 / 模块×责任端交叉矩阵
- 单条 LLM 责任分析仍保留，无 Key 时降级为本地 5Why 框架

### 6.1 LangGraph 节点设计（按 V1 能力）

```python
from langgraph.graph import StateGraph

class DefectAnalysisState(TypedDict):
    raw_defects: List[Dict]       # 原始台账
    cleaned_defects: List[Dict]   # 清洗后
    quality_report: Dict          # 质量报告
    clusters: List[Dict]          # 聚类结果
    trend_report: str             # 趋势报告
    final_report: str             # 最终 Markdown

graph = StateGraph(DefectAnalysisState)
graph.add_node("data_quality",  data_quality_node)    # 优先跑
graph.add_node("normalize",     normalize_node)        # 字段标准化
graph.add_node("cluster",       cluster_node)          # 聚类
graph.add_node("trend",         trend_node)            # 趋势
graph.add_node("compose",       compose_report_node)   # LLM 汇总

graph.set_entry_point("data_quality")
graph.add_edge("data_quality", "normalize")
graph.add_edge("normalize",    "cluster")
graph.add_edge("normalize",    "trend")  # 并行
graph.add_edge("cluster",      "compose")
graph.add_edge("trend",        "compose")
graph.set_finish_point("compose")
```

### 6.2 数据库表设计

```sql
-- 缺陷主表（统一标准模型）
CREATE TABLE defects (
    id VARCHAR(32) PRIMARY KEY,
    occurrence_time TIMESTAMP,
    defect_name TEXT NOT NULL,
    module VARCHAR(64),
    priority VARCHAR(8),                -- P0/P1/P2/P3
    problem_type VARCHAR(64),
    root_cause TEXT,
    discovery_channel VARCHAR(32),
    issue_owner VARCHAR(64),
    dev_owner VARCHAR(64),
    test_owner VARCHAR(64),
    responsibility_ratio VARCHAR(16),   -- 强制 "a:b" 格式
    has_loss BOOLEAN,
    fix_status VARCHAR(16),
    source_raw JSONB,                   -- 保留原始 JSON
    cleaned_at TIMESTAMP,
    created_at TIMESTAMP DEFAULT NOW()
);

-- 聚类标签（多对多）
CREATE TABLE defect_clusters (
    cluster_id VARCHAR(32),             -- 如 'copy_new_pattern'
    cluster_name VARCHAR(128),
    cluster_type VARCHAR(32),           -- 'business' | 'technical' | 'security'
    description TEXT,
    PRIMARY KEY (cluster_id)
);

CREATE TABLE defect_cluster_map (
    defect_id VARCHAR(32) REFERENCES defects(id),
    cluster_id VARCHAR(32) REFERENCES defect_clusters(cluster_id),
    confidence DECIMAL(3,2),            -- 0.00 ~ 1.00
    PRIMARY KEY (defect_id, cluster_id)
);

-- 数据质量问题记录
CREATE TABLE quality_issues (
    id SERIAL PRIMARY KEY,
    defect_id VARCHAR(32),
    field_name VARCHAR(64),
    issue_type VARCHAR(32),             -- missing / polluted / inconsistent
    detail TEXT,
    suggestion TEXT,
    detected_at TIMESTAMP DEFAULT NOW()
);
```
